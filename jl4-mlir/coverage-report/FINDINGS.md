# M2 — Compile-coverage triage findings

Generated by `scripts/coverage-sweep.py` over **438 `.l4` files** in the repo
(`jl4-mlir wasm --mlir-only`: typecheck + lower + schema, no LLVM toolchain).

## Outcomes

| Outcome           | Count | Meaning                                                             |
| ----------------- | ----: | ------------------------------------------------------------------- |
| `clean`           |    35 | all `@export`-ed functions lowered, every one `supported: true`     |
| `has-unsupported` |     2 | compiled, but ≥1 export flagged `supported: false` (M1a)            |
| `no-exports`      |   349 | typechecks but has no `@export` (course material, tests, fragments) |
| `typecheck-fail`  |    51 | jl4-core rejected it (a language/parse issue, not a backend gap)    |
| `lower-fail`      |     1 | misclassified — see below                                           |

## Headline conclusions

1. **The backend has essentially zero silent-degradation gaps.** Across the
   whole corpus only **2 functions** hit an unsupported construct — both the
   same `Eligible-Service-Requirement` deontic REGULATIVE rule (in the two
   copies of `ceo-performance-award.l4`). Both are now correctly flagged
   `supported: false` by M1a rather than silently returning FALSE. There were
   **no other unsupported-construct hits anywhere.**

2. **No real lowering crashes.** The single `lower-fail`
   (`jl4/examples/not-ok/tc/scombinator-fragment.l4`) is a deliberately-broken
   fixture with an indentation/parse error — my classifier just didn't match it
   to a typecheck bucket. There are **no genuine backend crashes** on the corpus.

3. **The "52 failures" are not backend gaps.** Categorised by path:

   | Category                                                    | Count |
   | ----------------------------------------------------------- | ----: |
   | `experiments/` (WIP/scratch)                                |    27 |
   | `not-ok/` + `tests-cli` garbage fixtures (designed to fail) |    23 |
   | **real files (genuine candidates)**                         | **2** |

   The only two "real" failures are parse-level edge cases unrelated to the MLIR
   backend:

   - `jl4/examples/lsp/semantic-tokens/directives.l4` (parse-error)
   - `jl4/ok/inert/simple.l4` (parse-error)

   Both fail at jl4-core's parse/typecheck stage, so they never reach lowering.

## Support / refuse decisions

- **Deontic / REGULATIVE** (the 2 hits): **refuse** at the backend (flag
  `supported: false`); route to a fallback evaluator. Full support is M6.
- **Lambda / higher-order**: refuse (flagged via M1a); the corpus has no
  exported instance, so it's latent. Revisit only if a real deployment needs it.
- **IO (FETCH/POST/ENV):** refuse; not meaningful for pure evaluation anyway.
- **Everything else that typechecks + exports:** supported today (35/37 of the
  real, exportable corpus lowers cleanly).

## Net

Of the **37 real, exportable files**, the backend handles **35 cleanly** and
**correctly flags 2** as needing the fallback. The coverage cliff implied by the
raw "52 failures" number is illusory — it's dominated by intentionally-broken
fixtures and experiments. The actionable backend coverage gap is just the
deontic family, already tracked as M6.
